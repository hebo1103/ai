# 项目测试准则 - 必须满足的条件

---

## 1. 准入阶段：提测质量红线（必须满足）

开发团队未满足以下条件，测试有权拒绝接收并直接打回。

- **静态扫描通过**：SonarQube 或对应工具检测无 Blocker/Critical 级别缺陷，新代码覆盖率 ≥ 80%。
- **单元测试通过率**：核心模块单元测试 100% 通过，失败用例必须已分析并记录原因。
- **冒烟测试自测**：开发需提供自测报告，主流程冒烟用例 100% 通过。
- **文档完备**：提供详细的 PRD/技术方案、数据库变更脚本、配置变更清单、接口文档（Swagger/YApi）。
- **依赖环境就绪**：测试环境数据库、中间件、第三方 Mock 服务已部署到位。

> **强制规则**：上述任一项不满足 → 测试直接打回，不计入测试工时，并发送邮件抄送项目经理和双方 leader。

---

## 2. 测试设计阶段：覆盖度硬性标准（必须满足）

- **需求全量追溯**：每条业务需求必须有对应的测试用例（正向、逆向、边界、异常），通过 QC/ALM/Jira 等工具建立需求-用例双向追溯矩阵。
- **覆盖率强制指标**：
  - 核心业务代码分支覆盖率 ≥ 95%
  - 普通业务代码分支覆盖率 ≥ 85%
  - 接口自动化脚本覆盖率 ≥ 90%（核心接口必须全部自动化）
- **安全与合规测试**：凡涉及用户隐私、支付、鉴权的模块，必须包含安全测试用例（SQL 注入、越权、敏感数据泄露等）。
- **异常与容灾场景**：至少包含：数据库断开重连、MQ 消息堆积、依赖服务超时/限流、主备切换等场景。

---

## 3. 测试执行阶段：准出条件（必须满足）

以下为上线前强制满足项，缺一不可。

### 3.1 功能测试

- **缺陷收敛趋势**：连续 3 个测试日内无新增 P0/P1 级缺陷。
- **缺陷修复率**：
  - **P0/P1（致命/严重）缺陷**：100% 修复并回归通过
  - **P2（一般）缺陷**：修复率 ≥ 95%，未修复的需在 Release Notes 中声明并得到产品/运营确认
  - **P3/P4（轻微/建议）缺陷**：需评估影响，上线不阻塞但需排期修复
- **回归测试全量通过**：所有回归测试用例（含自动化）100% 通过，无人为跳过。

### 3.2 非功能测试（必须执行）

- **性能测试**：核心接口需满足：TPS ≥ 目标值（如 1000）、TP99 ≤ 200ms、CPU/内存水位 ≤ 70%，并输出性能测试报告。
- **压力与稳定性测试**：1 小时持续压测，无内存泄漏、服务不重启、无 full GC 频繁问题。
- **兼容性测试**：覆盖主流浏览器/OS/设备组合（具体范围依据项目定，但必须书面明确）。
- **可靠性测试**：至少完成一次完整的故障注入演练（如 Kill 一个 Pod、从库延迟等）。

### 3.3 自动化与持续集成

- **CI 门禁**：每次代码合并前，单元测试 + 接口自动化 + 代码扫描必须全绿通过。
- **自动化测试通过率**：接入 CI 的自动化套件整体通过率 = 100%，不允许 flaky 用例未处理。

---

## 4. 测试报告与评审（必须满足）

**测试报告必须包含**：

- 测试范围与风险矩阵
- 缺陷统计（按模块、级别、类型）
- 未解决问题清单（含影响评估与应对措施）
- 质量门禁逐项结果（是否满足本准则第 2、3 条）

> **上线评审会议（必须）**：测试负责人拥有一票否决权。如果任一强制条件未满足，会议结论直接为"不通过"。

---

## 5. 发布后验证与复盘（必须满足）

- **发布后线上验证**：灰度/全量发布后 2 小时内，测试需执行核心场景冒烟用例，并实时监控业务核心指标（下单成功率、支付错误率等）。
- **线上故障回溯**：任何发布导致的 P0/P1 线上问题，必须在 3 个工作日内完成复盘报告，并补充对应的测试用例到回归集中。
- **准出后遗漏缺陷扣分机制**：若上线后一周内发现 P0/P1 级缺陷且属于测试范围内遗漏，计入测试质量扣分（与团队绩效挂钩）。

---

## 6. 必须满足的核心检查清单（测试负责人每日/每阶段自检）

| 阶段 | 强制条件 |
|------|----------|
| **测试准入** | □ 提测质量门禁通过 □ 冒烟用例 100% 通过 |
| **测试设计** | □ 需求追溯矩阵完成 □ 覆盖率达标 □ 安全用例覆盖 |
| **测试执行** | □ 缺陷收敛 □ 性能测试通过 □ 兼容性测试通过 |
| **测试准出** | □ P0/P1 缺陷清零 □ 回归测试全绿 □ 报告评审通过 |
| **上线后** | □ 线上验证完成 □ 关键指标监控正常 |
---

## 附：违反准则的处理机制（必须执行）

- **测试人员违反**：如未执行强制项（如性能测试）就签署准出，取消当季度评优资格，并进行质量回溯。
- **开发/产品违反**：跳过准入门禁强行要求测试、或上线评审无视一票否决权，由项目经理发起质量事故告警，计入组织绩效扣分。
- **项目豁免**：任何需要豁免以上某项条件的项目，必须由项目负责人、技术负责人、测试负责人三方签字同意，并归档豁免单。
